Voice Agent Service

Browser Sessions and Chat-Pipeline Voice Agents

Browser clients can now join LiveKit rooms backed by JAI chat-pipeline voice agents. The path reuses the existing agent_service runner, agent tools, prompt context, and event stream instead of maintaining a separate voice-only agent stack.

Executive Summary

Value

One voice path for browser apps

Any approved browser client can request a short-lived LiveKit session through the browser-session API and connect to a dispatched voice agent.

Existing agent investments carry over

The voice worker delegates turns into the same chat-pipeline runner, preserving agent configuration, tools, project/artifact context, and prompt behavior.

Observable and controllable sessions

The API reserves and finalizes sessions, enforces active-session limits, emits BigQuery lifecycle events, and cleans up stale LiveKit dispatches.

End-to-End Architecture

Flow
Control/API path LiveKit media and dispatch path Browser event path Storage/observability side effects

Control plane

The public API authenticates, resolves prompts, reserves sessions, creates LiveKit rooms, dispatches workers, and finalizes sessions.

Streaming output plane

agent_service streams text and tool events. voice_service maps text into ChatChunk.delta.content for LiveKit TTS audio, and maps structured payloads into jai.agent_events for browser data events.

Slide 4

jai-platform/voice_service

Runtime

LiveKit worker responsibilities

  • Runs a LiveKit AgentServer entrypoint for the configured voice agent worker name.
  • Parses dispatch metadata into a typed DispatchPayload.
  • Builds STT, TTS, VAD, turn detection, endpointing, and interruption behavior.
  • Starts AgentSession with room IO and handles lk.chat text input.
  • Collects LLM, STT, and TTS usage metrics.

Runner bridge responsibilities

  • Creates an agent_service runner from room, org, user, agent, project, artifact, tool, and custom-instruction metadata.
  • Adapts runner event chunks into LiveKit ChatChunk responses for speech output.
  • Publishes structured runner events to jai.agent_events.
  • Persists assistant text, tool calls, tool outputs, errors, and chat history.
Key implementation point: prewarm initializes the app runner and VAD early, while each LiveKit job creates a room-scoped runner so voice turns execute through the same agent path as chat.

Slide 5

Why Reuse agent_service Runner

Advantages

Tool parity

Voice sessions can call the tools already configured on the agent instead of defining a separate voice tool registry.

Prompt parity

Custom instructions, system prompt behavior, and existing chat-pipeline prompt composition are preserved.

Context parity

Project IDs, artifact IDs, agent labels, orgs, users, and room IDs are passed through a single runner context.

Event parity

Text chunks, tool calls, tool outputs, and errors follow the same event model and can be forwarded to the browser.

Standalone voice stack risk Separate prompts, separate tools, separate event handling, and divergent behavior from chat agents.
Runner-backed voice path One agent implementation can serve chat and browser voice sessions with consistent capabilities and observability.

Slide 7

Public Browser-Session Interface

Contract
Surface Details
POST /voice-agent/v1/browser-sessions Creates a tracked LiveKit browser session. Requires application API key. Important fields: agentId, optional prompt, optional userId, language, projectId, artifactIds, and participantName.
POST response sessionId, token, livekitUrl, generated roomName, and LiveKit dispatchId returned from createDispatch.
DELETE /voice-agent/v1/browser-sessions/:id Finalizes the tracked browser session, records end time and duration, and performs best-effort LiveKit dispatch teardown.
Throttling and quota The service enforces an active browser-session limit per org and treats sessions older than the max duration as stale.

For chat_pipeline, prompt is forwarded as custom_instruction. For legacy realtime or pipeline voice configs, prompt remains required.

Slide 8

Reusable Studio Page Pattern

Adoption

Other teams can create their own browser voice sessions from Studio Pages by adding a small server-side session route that wraps the public browser-session API, then connecting the page UI to LiveKit with the returned credentials.

Configure the agent in Japan AI client. Enable chat_pipeline voice on the agent, choose TTS/STT/language/eagerness settings, and attach tools through the normal agent configuration.
Add a Studio Page server route. Keep JAPAN_AI_VOICE_API_KEY server-side. The browser should call the page route, not the public API directly.
Create a browser session. The route calls POST /voice-agent/v1/browser-sessions with agentId, optional prompt, userId, language, projectId, artifactIds, and participantName.
Connect the browser to LiveKit. The page receives sessionId, token, livekitUrl, roomName, and dispatchId, then joins the room with the LiveKit client.
Handle voice events. Use lk.chat for text input, lk.transcription for transcript segments, lk.agent.state for turn UI, and jai.agent_events for text/tool/error payloads.
Finalize the session. Call DELETE /voice-agent/v1/browser-sessions/:id from the server route on end, pagehide, timeout, or failure cleanup.

What teams own

  • Page authorization and tenant/session guardrails.
  • Custom instruction source, such as page context, workflow state, or uploaded prompt content.
  • Browser UI for microphone state, transcript rendering, tool events, and completion behavior.

What the platform owns

  • LiveKit room creation, agent dispatch, token issuance, active-session limits, and cleanup.
  • Room metadata handoff into voice_service and agent_service.
  • Forwarding configured agent tools through agent_tools for the voice runner.

Slide 9

Chat-Pipeline Dispatch Metadata

LiveKit

The browser-session API dispatches the chat voice worker with metadata that gives the voice worker everything it needs to reconstruct the chat-pipeline runner context, including the LiveKit room details.

org_id application_id user_id agent_id agent_label agent_tools project_id artifact_ids room_id language custom_instruction tts tts_provider voice_id stt_id stt_language allow_interruption eagerness
Room handoff: browser-session generates roomName, calls LiveKit createRoom(roomName), then calls createDispatch(roomName, ...), which returns dispatchId. The API returns both roomName and dispatchId to HR/browser, and sends the same room value in dispatch metadata as room_id. The browser joins with the returned token and room details; the agent worker receives the LiveKit job for that room and starts AgentSession with ctx.room.

Legacy realtime/pipeline

Dispatches the older LiveKit agent path and requires a prompt for session creation.

New chat_pipeline

Dispatches jai-chat-voice-agent-{env}, forwards the agent tool list, and maps the browser-session prompt to runner custom_instruction.

Schema and UI Enablement

Config

Agent schema

  • Adds chat_pipeline voice-call config.
  • Supports chat TTS provider and voice selection across Deepgram, OpenAI, and ElevenLabs.
  • Supports STT model, STT language, interruption, and eagerness endpointing metadata.
  • Normalizes provider-specific TTS payloads before dispatch.

Admin and client UI

  • Admin schema validation accepts valid chat-pipeline config and rejects unsupported TTS config.
  • Client settings expose pipeline type, TTS provider, voice, STT, language, eagerness, and interruption settings.
  • Teams can configure agents through normal settings UI instead of bespoke HR-only code.
This is the productization step: browser-session voice agents become configurable platform capability, not a one-off integration.

Slide 11

HR Interview Integration

Prompt and session
Candidate page requests session. HR calls its own /api/realtime-session, protected by a capability ticket with session scope.
Prompt is resolved server-side. Position prompt is preferred, then org prompt, then fallback default prompt.
Prompt file content is forwarded whole. HR downloads prompt content via PRS storage proxy and sends it as browser-session prompt.
Public API maps prompt to voice context. For chat-pipeline agents, that prompt becomes LiveKit metadata custom_instruction.
Browser receives only session details. The response includes token, URL, room/session identifiers, and a coarse prompt-source label, not the prompt content.
Required HR env JAPAN_AI_VOICE_API_URL, JAPAN_AI_VOICE_API_KEY, JAPAN_AI_VOICE_AGENT_ID, JAPAN_AI_VOICE_USER_ID
Optional HR env Voice language and artifact IDs; prompts are read through the PRS storage proxy.

Slide 12

HR LiveKit Client Behavior

Browser

Connection and turn control

  • Connects to LiveKit with the browser-session token and URL.
  • Publishes the candidate microphone muted at first.
  • Waits for the agent participant to join.
  • Sends the first lk.chat text message to trigger the opening statement.
  • Uses agent state to decide when the candidate can speak.

Events parsed by HR

  • lk.transcription updates user and agent transcript segments.
  • lk.agent.state drives listening, thinking, speaking, avatar, and mic state.
  • jai.agent_events carries runner text, tool calls, tool outputs, and errors.
  • LiveKit audio tracks drive candidate playback, agent speaking state, and lip-sync behavior.
The first text message is deliberate: HR asks the agent to use <OpeningStatement> when present, then proceeds with normal voice interaction.
